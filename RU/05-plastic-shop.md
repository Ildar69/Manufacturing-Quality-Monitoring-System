## Ключевое отличие: другая логика ввода

В цехе кузовов оператор пишет `1 2` — код дефекта и количество. Скрипт нормализует.

Здесь всё иначе. Оператор работает с выпадающими списками которые генерируются динамически в зависимости от контекста. Не нужно знать коды, не нужно помнить что такое `1` или `5`. Выбираешь модель — список сужается. Выбираешь деталь — видишь только то что относится к этой модели.

Это осознанное решение. Другой цех, другие люди, другой уровень привычки к цифровым инструментам. Интерфейс должен был быть проще.

## Умный ввод: как работают динамические списки

Структура строки:

|Столбец|Данные|
|-|-|
|A|Порядковый номер|
|B|Дата и время — проставляется автоматически при вводе SKID|
|C|SKID|
|D|Модель — выпадающий список|
|E|Цвет — генерируется от модели|
|F|Наименование детали — генерируется от модели|
|G|Дефект|
|H|Количество|
|+|Слой покрытия, метод устранения, статус|

**Одна строка — один дефект на одной детали.** Это принципиальное отличие от цеха кузовов где одна строка — один кузов со всеми дефектами.

### Динамические списки — почему это важно

Когда оператор выбирает модель — скрипт смотрит в именованные диапазоны (`COL\_{model}` для цветов, `\_{model}` для деталей) и строит выпадающий список только из значений этой модели. У одной модели 3 цвета, у другой 10 — оператор видит только своё.

```javascript
function setupRowValidation(ss, sheet, row, model) {
  var namedRanges = ss.getNamedRanges();
  for (var i = 0; i < namedRanges.length; i++) {
    var name = namedRanges\[i].getName();
    if (name == "COL\_" + model) // цвета этой модели
    if (name == "\_"   + model)  // детали этой модели
  }
}
```

### Автокопирование строки

Если оператор добавляет новый дефект на тот же кузов — скрипт автоматически копирует SKID, модель, цвет, деталь и выпадающие списки из предыдущей строки. Не нужно вводить всё заново — только дефект и количество.

### Автоматический статус

При выборе метода устранения (ОК / Ремонт / Доработка на линии) — в служебный столбец автоматически проставляется машиночитаемый статус (`ok` / `not ok`). Это нужно для корректного подсчёта в аналитике.

## Глубокая аналитика — детальный отчёт по запросу

Отдельный файл для каждой линии. Выбираешь период — получаешь полный отчёт. 850+ строк JS, версия v7.

**Логика подсчёта скидов — нетривиальная:**

Один кузов может появляться несколько раз подряд (несколько дефектов) а потом появиться снова после другого кузова. Это разные визиты — разные скиды. Простой подсчёт уникальных значений даст неверный результат.

Решение: последовательная группировка. Один скид = непрерывная последовательность строк с одним VIN в рамках дня. Если тот же VIN появился после перерыва — это новый скид.

```
060 → скид 060\_d\_1
060 → скид 060\_d\_1  (подряд — тот же)
020 → скид 020\_d\_1
060 → скид 060\_d\_2  (прерывался — новый)
Итого: 3 скида, не 2
```

**Что внутри отчёта:**

1. Общая статистика: всего скидов, ОК, ремонт, доработка на линии, DPR, DPR-2, DPU
2. Статистика по моделям
3. DPR по дням
4. DPU по дням в разрезе каждой модели
5. Динамика дефектов по дням — с почасовым и модельным разбором
6. Все дефекты за период по убыванию
7. DPU дефектов по моделям — сравнительная таблица
8. Детализация по деталям: деталь → дефект → количество → DPU
9. Дефекты по статусу устранения: что уходило в ремонт, что дорабатывалось на линии
10. Сводная матрица: дефект × статус

Два показателя DPR считаются по-разному:

* **DPR** = OK / Всего × 100 — только чистый проход
* **DPR-2** = (Всего − Ремонт) / Всего × 100 — включая доработку на линии

Условное форматирование: DPU ≥ 0.3 — красный, 0.1–0.3 — жёлтый, < 0.1 — зелёный. Проблемы видны сразу без чтения цифр.

## Про Microsoft stack

Рассматривал возможность развернуть систему на Microsoft stack. Изучил — отказался. Инструменты не дают той гибкости которая нужна для подобных задач, а обходные решения превращаются в трудно поддерживаемые конструкции. Google Workspace в данном контексте оказался правильным выбором.

|Блоки|Цех кузовов|Цех пластика|

|Инициатива|Логичное развитие роли|Полностью собственная|
