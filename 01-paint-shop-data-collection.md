## С чего всё началось

Дефекты фиксировались на бумаге. Только на одной линии из шести. Один лист на один кузов, заполнялся вручную несколькими людьми. В конце смены — стопка бумаги, которую никто системно не анализировал.

Остальные пять линий не вели никакой записи вообще.

Моя роль не подразумевала разработку систем. Я занимался статистикой. Но я видел что данные есть — их просто никто не собирает. Я решил это исправить.

## Первое решение: не один файл, а система файлов

Первый вопрос который я решал: как организовать сбор данных с 6 линий? Один общий файл — очевидное решение. Я от него отказался.

Причина: каждая линия — это разный набор данных. На одной фиксируется статус кузова, на другой — цвет покрытия, на третьей — нет некоторых элементов которые есть на остальных. Один универсальный файл означал бы компромиссы везде: лишние столбцы, путаница, сложнее объяснять операторам.

Решение: 4 отдельных файла. У каждого свой Apps Script, своя структура, своя логика. Файлы точно отражают реальность своей линии — не больше и не меньше.

Три линии получили по отдельному файлу. Ещё три — финальный контроль, камера ремонта и выходной контроль — объединены в один файл с разными листами, потому что они технологически связаны: данные между ними движутся автоматически.

| Линия | Файл | Ключевая особенность |
|---|---|---|
| Входящий контроль | Отдельный | Базовый набор: идентификатор + модель + дефекты по элементам |
| Промежуточный контроль 1 | Отдельный | То же + статус кузова → источник метрики DPR |
| Промежуточный контроль 2 | Отдельный | То же + дополнительный параметр покрытия |
| Финальный контроль | Общий файл, лист 1 | Полная структура: 5 полей идентификации + 26 элементов + статус + шифр ремонта |
| Камера ремонта | Общий файл, лист 2 | Получает данные через QUERY, расшифровывает шифр по элементам |
| Выходной контроль | Общий файл, лист 3 | Объединяет два потока: прямой выход и выход после ремонта |

## Нормализация: как сделать данные пригодными для анализа

Операторы — не аналитики. Они вводят данные как удобно: с пробелом, запятой, слешем, точкой. Без нормализации аналитика невозможна: один и тот же дефект записан в десяти форматах — система не может его посчитать.

Я разработал единый паттерн нормализации и реализовал его в каждом файле. Формат: `код_дефекта(количество)`.

```
Оператор вводит:  2 3    или  2,3   или  2/3
Скрипт пишет:     2(3)

Несколько дефектов на одном элементе:
Ввод:   1 2 5 1
Итог:   1(2),5(1)
```

Дополнительно — нормализация идентификатора кузова. Оператор вводит просто число, скрипт добавляет префикс и дополняет до нужного формата:

```
125    →  BD125
bd125  →  BD125
```

### Два уровня нормализации — почему не один

Первый очевидный вариант — нормализовать при вводе (`onEdit`). Проблема: стандартный `onEdit` в Google Apps Script не имеет доступа к `PropertiesService` и `LockService`. При быстром вводе или одновременном редактировании с нескольких устройств — данные могут повредиться.

Решение: два уровня.

- **Уровень 1 — `onEdit`:** мгновенная нормализация при вводе. Пользователь видит результат сразу. Если формат неверный — скрипт откатывает значение и показывает уведомление.
- **Уровень 2 — `periodicSync` (каждую минуту):** гарантия корректности. Перепроверяет все строки, досчитывает суммы, восстанавливает метки времени. Работает через `LockService` — только один экземпляр в момент времени.

Дополнительно: установлен `onEditInstallable` — копия триггера с полным доступом к сервисам. Оба вызывают одну функцию `_handleEdit()`. Это обходит ограничение стандартного `onEdit`.

## Конкурентный доступ: проблема которую не ждёшь

На линии работают несколько операторов одновременно. Каждый — со своего планшета. Все пишут в один файл. Google Sheets не рассчитан на такой сценарий из коробки.

Проблема проявляется так: два оператора редактируют разные строки одновременно — срабатывают два `onEdit` — оба пытаются записать метку времени — один перезаписывает другого или получает ошибку блокировки.

Решение — write-ahead log через `ScriptProperties`:

1. При первом редактировании триггерного столбца — время сохраняется в `ScriptProperties` под ключом `TS_A_{row}`
2. `periodicSync` забирает это значение и переносит в ячейку под защитой `LockService`
3. После успешной записи — ключ из Properties удаляется

Это гарантирует: точное время первого ввода не теряется даже при гонке между несколькими `onEdit`.

## Финальный контроль: самый сложный файл

Это центральный файл системы. Здесь больше всего данных, самый большой скрипт, три взаимосвязанных листа. Именно здесь до системы велась единственная бумажная запись.

### Структура записи

Каждая строка — один кузов. Пять полей идентификации, затем дефекты по 26 элементам, затем статус и данные для ремонта:

| Столбец | Данные |
|---|---|
| A | Идентификатор кузова (VIN) |
| B | Номер скида — нормализуется автоматически |
| C | Модель |
| D | Цвет |
| E | Дополнительный параметр комплектации |
| F:AE | Дефекты по 26 элементам кузова |
| AF | Статус: ОК / Ушел на ремонт |
| AJ | Шифр дефектов при отправке на ремонт |
| AK | Комментарий произвольным текстом |

### Шифр дефектов — моё решение

Задача: оператор должен быстро передать информацию о том, что именно надо отремонтировать — по какому элементу, какой дефект. Текстом — долго и неоднозначно. Таблицей — неудобно вводить.

Я придумал компактный шифр: 26 элементов кузова закодированы буквами, дефекты — числами. Запись строится как список пар:

```
A1,2,B3

A = первый элемент  →  1 = первый тип дефекта, 2 = второй тип дефекта
B = второй элемент  →  3 = третий тип дефекта
```

Оператор не пишет словами. Работники ремонта читают шифр и уже знают что чинить — до того как кузов физически к ним приехал. Подготовка начинается заранее.

## Камера ремонта: автоматическая расшифровка

Отдельный лист в том же файле. Данные попадают сюда автоматически через QUERY — без участия человека:

```
=QUERY(ARRAYFORMULA(TO_TEXT(Информация)); "select * where Col32 = 'Ушел на ремонт'"; 1)
```

Дальше — расшифровка шифра. Воспроизведена та же структура из 26 элементов. Под каждым — формула, которая смотрит в столбец AJ этого кузова и извлекает нужные числа:

```
=ЕСЛИОШИБКА(
  REGEXREPLACE(REGEXREPLACE(
    REGEXEXTRACT($AJ4 & " "; "(?i)" & AU$2 & "([\d\s,]+)");
  "[^\d,]+"; ""); "^,|,$"; ""); "")
```

Результат: работник ремонта видит таблицу — элемент, номера дефектов. Не нужно читать и интерпретировать шифр вручную. После ремонта: фиксирует что сделано, проставляет время, выбирает статус выезда.

## Выходной контроль: два потока в одном листе

Финальная точка. В лист попадают кузова из двух источников:

- Кузова со статусом ОК из финального контроля — прошли без отклонений
- Кузова со статусом "Выехал" из камеры ремонта — прошли после доработки

Скрипт `refreshOKSheet` обновляется каждые 5 минут. Объединяет оба источника, дедуплицирует по составному ключу `VIN+SKID+модель`, сохраняет комментарии и отметки операторов между обновлениями.

Скользящее окно — последние 3 дня в живом листе. Параллельно ведётся архивный лист: данные только добавляются, никогда не удаляются. При обновлении живого листа — комментарии из архива восстанавливаются, чтобы не потерять работу операторов.

## Поток данных через систему

```
[Входящий контроль]          → отдельный файл, данные не передаются дальше
[Промежуточный контроль 1]   → отдельный файл, статус кузова → метрика DPR
[Промежуточный контроль 2]   → отдельный файл
[Финальный контроль]  ──────────────────────────────────┐
    ↓ QUERY (статус = ремонт)                           │
[Камера ремонта]                                        │ один файл
    ↓ статус = выехал                                   │
[Выходной контроль] ← также ← [статус ОК из финала]   │
    ↓                                                   │
[Архив] — только дозапись                              ─┘
```

Данные движутся в одном направлении — вниз по процессу. Каждый файл знает только своё. Аналитика собирает всё вместе — но это отдельный блок.

## Решения которые пришлось принять

| Задача | Варианты | Принятое решение |
|---|---|---|
| Организация данных по 6 линиям | Один общий файл vs отдельные файлы | 4 отдельных файла — каждый точно отражает свою линию, без компромиссов в структуре |
| Нормализация свободного ввода | Валидация vs нормализация | Нормализация: любой ввод приводится к формату, не отвергается |
| Конкурентный доступ | Игнорировать vs решать | Write-ahead log через ScriptProperties + LockService в periodicSync |
| Метка времени при быстром вводе | Время из onEdit vs отдельное хранение | Сохранение в Properties при первом срабатывании, перенос в ячейку через periodicSync |
| Передача информации на ремонт | Текст словами vs структурированный шифр | Компактный шифр: буква = элемент, цифра = дефект — быстрый ввод, автоматическая расшифровка |
| Актуальность данных в листе ОК | Полная история vs скользящее окно | Скользящее окно 3 дня + отдельный архив без удалений |

## Что это на самом деле

Это не набор таблиц. Это распределённая система сбора данных с несколькими точками ввода, автоматической нормализацией, защитой от конкурентного доступа и связанными потоками данных между файлами.

Каждый из 4 файлов — самостоятельный модуль со своим скриптом. Три из них связаны между собой через QUERY и формулы. Всё работает в браузере, без установки ПО, на планшетах прямо на производстве.

Система написана одним человеком за месяц. В продакшне — 4+ месяца. Внедрялась в живой цех: обучение операторов, компромиссы с руководством, итерации под реальное использование.
