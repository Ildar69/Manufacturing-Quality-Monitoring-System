# Цех окраски кузовов — аналитика

*Два уровня: реальное время · глубокий анализ по запросу*

\---

## Два уровня — две разные задачи

Аналитика в этой системе разделена на два независимых уровня, и это не случайно.

**Реальное время** — отвечает на вопрос "что происходит прямо сейчас". Графики обновляются автоматически, без участия человека. Задержка между вводом данных оператором и появлением на дашборде — секунды.

**Глубокая аналитика** — отвечает на вопрос "что происходило за период". Запускается вручную: выбираешь линию, задаёшь даты, получаешь детальный отчёт. Нужна когда надо понять тренд, сравнить периоды, разобрать конкретную проблему.

Это разные инструменты для разных вопросов. Совмещать их в одном месте — значит делать оба хуже.

\---

## Архитектура: почему не один файл

Первое решение которое пришло бы в голову — собрать всю аналитику в один файл. Я от него отказался.

Причина практическая: центральный файл с данными по всем линиям уже сейчас открывается только на десктопе. На мобильном устройстве Google Sheets вылетает — слишком много данных, приложение не справляется. Один файл на всё означал бы единую точку отказа: сломался — нет ничего.

Решение: каждый аналитический файл делает одну конкретную задачу. Если один файл недоступен — остальные продолжают работать. Данные есть всегда.

Итого в аналитическом слое: 10+ отдельных файлов, каждый со своим назначением.

\---

## Слой 1: централизация данных

Первый шаг — собрать данные из всех линий в одном месте.

Каждая исходная таблица (куда вносят операторы) отдаёт данные через IMPORTRANGE + QUERY напрямую — в центральный файл и в аналитические таблицы параллельно. Центральный файл не является единственным получателем: данные расходятся из источника веером.

```
\\\[Таблица линии — ввод данных]
        ↓ IMPORTRANGE + QUERY
        ├──→ \\\[Центральный файл — все линии]
        ├──→ \\\[Аналитика: топ дефектов]
        ├──→ \\\[Аналитика: по сторонам]
        └──→ \\\[Аналитика: по моделям]
```

Пример запроса который тянет данные из исходной таблицы:

```
=QUERY(
  {
    ARRAYFORMULA(TO\\\_TEXT(IMPORTRANGE("source\\\_id"; "'Sheet'!A4:E")))\\\\ 
    IMPORTRANGE("source\\\_id"; "'Sheet'!F4:AK")
  };
  "select \\\* where Col1 is not null";
  0
)
```

Логика разделения на TO\_TEXT и числовые данные: идентификаторы (VIN, модель, цвет) нужны как текст чтобы QUERY мог с ними работать корректно. Числа и даты идут отдельно чтобы сохранить типы для расчётов.

\---

## Слой 2: расчёты

Между исходными данными и графиками — слой расчётов. Отдельные таблицы считают метрики: DPU по дням, DPR, распределение по элементам, топ дефектов за сегодня, сравнение сторон.

Эти расчёты идут обратно в центральный файл — в отдельные листы, на которые настроены графики.

```
\\\[Центральный файл — сырые данные]
        ↓
\\\[Таблицы расчётов — метрики]
        ↓
\\\[Центральный файл — листы с данными для графиков]
        ↓
\\\[Графики — строятся автоматически]
```

Всё автоматически. Я ничего не запускаю руками. Данные появились — расчёты пересчитались — графики обновились.

\---

## Слой 3: дашборды реального времени

В центральном файле три дашборда — три листа с графиками под разные вопросы.

**Дашборд 1 — общая картина по линии:**

* DPU, количество единиц, прямой выход, ремонт — в разрезе дней
* DPR в разрезе дней
* Топ 10 элементов по количеству дефектов — за сегодня
* Топ 10 видов дефектов — за сегодня
* Сравнение правой и левой стороны по ключевым дефектам — за сегодня

**Дашборд 2 — разрез по моделям:**

* Количество единиц по каждой модели — таблица
* Топ 10 дефектов отдельно для каждой модели — за сегодня

**Дашборд 3 — симметрия дефектов по моделям:**

* Для каждой модели — отдельный график сравнения правой и левой стороны
* Позволяет увидеть: дефект системный (обе стороны) или локальный (одна сторона)

Все три дашборда обновляются сами. Каждый день — актуальные данные без каких-либо действий с моей стороны.

## Защита кода: библиотеки Apps Script

Вся логика расчётов вынесена в Apps Script Libraries. В каждом аналитическом файле в редакторе скриптов — только вызовы библиотеки:

```javascript
MyLib.generateReport();
MyLib.updateDashboard();
```

Сама реализация — в библиотеке, к которой нет публичного доступа. Открыть файл и прочитать логику невозможно — виден только вызов.

Доступ к файлам — стандартный Google: по приглашению на конкретный email. Случайный человек не может открыть ни один из файлов системы.

## Слой 4: глубокая аналитика по запросу

Отдельный инструмент — для детального разбора за произвольный период.

**Как работает:**

1. Открываешь файл, видишь Dashboard — лист с таблицей управления
2. Напротив нужной линии вводишь даты "С" и "ПО"
3. Через меню: Аналитика → линия → Обновить базу
4. Затем: Аналитика → линия → Отчёт: General или По моделям
5. Получаешь готовый лист с полным разбором

**Что внутри отчёта General:**

* Сводные метрики: количество единиц в базе, всего дефектов, DPU, ушли в ремонт, DPR
* Дефекты по элементам: сколько дефектов на каждом элементе, DPU по элементу
* Разрез по видам дефектов: номер, название, количество, DPU, правая сторона, левая сторона, разница правая-левая, разбивка по каждой паре элементов и непарным элементам

**Что внутри отчёта По моделям:**

* Те же блоки, но отдельно для каждой модели
* Можно сравнить: одна модель проблемная или проблема системная

**Как устроен код:**
850+ строк JS. Конфигурация вынесена в объект `CONFIGS\\\_` — под каждую линию свои параметры: откуда брать данные, какие столбцы, какие элементы кузова, какие дефекты, есть ли DPR. Добавить новую линию — значит добавить блок конфигурации, логика переиспользуется.

```javascript
const CONFIGS\\\_ = {
  Line1: {
    sourceId:    "...",
    defectStart: 3,
    defectEnd:   25,
    hasDpr:      false,
    elements:    \\\[...],
    defectNames: {...}
  },
  Line2: { ... }
};
```

Парсинг данных, построение отчётов, условное форматирование (правая сторона оранжевая, левая синяя, разница подсвечивается) — всё генерируется программно из конфигурации.

## Что это даёт на практике

Реальное время отвечает на вопрос прямо сейчас: сколько дефектов, на каком элементе, по какой модели, какая сторона проблемная сегодня.

Глубокая аналитика отвечает на вопрос за период: что изменилось за месяц, какой дефект стал чаще, какая модель хуже других, есть ли системная проблема по одной стороне.

Именно комбинация двух уровней даёт полную картину — оперативную и историческую одновременно.
